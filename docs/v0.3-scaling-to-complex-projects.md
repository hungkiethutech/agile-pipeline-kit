# v0.3 — Scaling to complex, many-module projects

Earlier the kit handled one feature/ticket well but had 3 gaps that hurt large, multi-module
projects. v0.3 adds a targeted fix for each.

| Gap (from the honest assessment) | v0.3 fix |
|---|---|
| **Context window** — agents can't hold the whole system; "read the artifact" isn't enough as the codebase grows. | **Persistent context layer** (`context/`): `architecture.md` (invariants + shared contracts), `module-map.md` (each module's ownership/contract/deps + "Do NOT" rules), `adr/` (decisions). Every agent reads it FIRST; it's small, curated memory — not full code. |
| **Cross-module consistency** — a linear per-ticket pipeline doesn't guard the seams between modules. | **Integration stage** (`ap-integration-agent`, stage 5b): auto-runs when a ticket touches 2+ modules; tests the dependency edges from `module-map.md` end-to-end (data consistency, transactions, contract-drift probes). |
| **Long-term memory between tickets** — decisions from ticket #1 don't reach ticket #50; drift. | **Architecture updates context after every ticket** (module-map + ADRs) **+ an `arch-review` gate** (`ap-arch-review-agent`) right after Dev that blocks any change violating invariants, module boundaries, or an undocumented shared-contract change. |

## New flow
```
requirements → (design ∥ architecture) → development
  → arch-review        (coherence gate; back to Dev on drift)
  → qa                 (black-box, per feature)
  → integration        (only if 2+ modules touched)
  → security → devops → ops
  → architecture updates context/ (module-map + ADR)   ← closes the memory loop
```

## What this does and does NOT change
- **Does:** make the kit far more coherent across many modules and many tickets; catch cross-module
  breakage and architectural drift early and cheaply.
- **Does NOT:** remove the human. The 2 approval gates stay, and for genuinely complex systems a human
  architect should still review the `context/` layer periodically. It also does not lift the
  subscription usage limits — big projects = many tickets = real token/time cost.

## Honest status
Still validate with a real pilot per project. v0.3 is a design-level upgrade; the context layer is
only as good as the discipline of keeping it current (the pipeline now enforces that, but review it).
